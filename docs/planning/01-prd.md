## 1. 문제 정의

### 1.1 현재 상황 (As-Is)

StoryFlow Creator는 완성도 높은 노코드 웹 빌더입니다. 하지만 모든 기능이 무료로 제공되어 서비스 운영비(Supabase, 이미지 저장소 등)를 지속적으로 충당하기 어렵습니다.

비유하자면, 맛있는 음식을 무료로 제공하는 식당과 같습니다. 손님은 많지만 재료비를 감당할 수 없는 상황입니다.

### 1.2 목표 상황 (To-Be)

- 무료 사용자도 핵심 기능을 충분히 사용할 수 있음
- 더 많은 기능이 필요한 사용자는 합리적인 가격에 유료 플랜 구독
- 월 운영비를 유료 구독 수익으로 충당
- 공격적인 업셀 압박 없이 자연스러운 결제 유도

### 1.3 왜 지금인가? (Why Now)

- 예방 차원: 사용자 증가 전에 수익 구조를 마련
- 지속 가능성: 장기적인 서비스 운영을 위한 기반 구축
- 현재 구현된 기능이 충분히 가치 있어 유료화 가능

## 2. 사용자 페르소나

### 2.1 주요 페르소나

| 항목 | 내용 |
|------|------|
| 이름/별칭 | "바쁜 민지" |
| 나이/직업 | 28세 프리랜서 디자이너 |
| 핵심 니즈 | 포트폴리오 페이지를 빠르게 만들고 싶음 |
| 불편 포인트 | 코딩을 모름, Wix는 광고가 많고 느림 |
| 사용 상황 | 클라이언트에게 포트폴리오 보여줄 때 |

### 2.2 페르소나의 하루 (시나리오)

**무료 사용자 시나리오:**
1. 민지는 StoryFlow로 포트폴리오 페이지를 만듦
2. 처음 3개 프로젝트는 무료로 충분히 사용
3. 4번째 프로젝트가 필요해지자 "프로젝트 한도 도달" 메시지 확인
4. 월 9,900원에 무제한 사용 가능하다는 안내 확인
5. "Notion처럼 필요한 만큼만 쓰면 되네" 하고 Pro 결제
6. 브랜딩 제거도 함께 되어서 더 전문적인 페이지 완성

## 3. 사용자 스토리

### FEAT-0: 공통 흐름 (기존 온보딩/로그인)
- [x] 사용자로서, 이메일/소셜 로그인으로 가입할 수 있어야 한다. (기존 구현됨)
- [x] 사용자로서, 로그인 상태가 유지되어야 한다. (기존 구현됨)

### FEAT-1: 플랜별 기능 제한 시스템 (MVP)
- [ ] 무료 사용자로서, 3개의 프로젝트를 만들 수 있어야 한다. 왜냐하면 기본적인 사용은 무료로 가능해야 하므로.
- [ ] 무료 사용자로서, 하루 10회 내보내기가 가능해야 한다. 왜냐하면 적절한 제한이 필요하므로.
- [ ] 무료 사용자로서, 1명의 협업자만 초대할 수 있어야 한다. 왜냐하면 팀 기능은 유료 가치가 있으므로.
- [ ] 무료 사용자로서, 내보낸 HTML에 "Made with StoryFlow" 워터마크가 표시되어야 한다.
- [ ] Pro 사용자로서, 프로젝트/내보내기/협업자가 무제한이어야 한다. 왜냐하면 유료 가치를 제공해야 하므로.
- [ ] Pro 사용자로서, 워터마크가 제거되어야 한다. 왜냐하면 전문적인 페이지가 필요하므로.

### FEAT-2: 유료 플랜 결제 시스템 (v2)
- [ ] 사용자로서, 토스페이먼츠로 월 구독 결제를 할 수 있어야 한다.
- [ ] 사용자로서, 언제든 구독을 취소할 수 있어야 한다.
- [ ] 사용자로서, 결제 내역을 확인할 수 있어야 한다.

### FEAT-3: Team 플랜 (v2 이후)
- [ ] Team 사용자로서, 팀원들의 권한을 관리할 수 있어야 한다.
- [ ] Team 사용자로서, 팀 전체 프로젝트를 한 곳에서 볼 수 있어야 한다.

## 5. 범위 관리

### 5.1 MVP (반드시 포함)
- FEAT-1: 플랜별 기능 제한 시스템
  - 프로젝트 개수 제한 (무료: 3개)
  - 내보내기 횟수 제한 (무료: 일 10회) - 기존 구현됨
  - 협업자 수 제한 (무료: 1명)
  - 브랜딩 워터마크 (무료: 표시, Pro: 제거)

### 5.2 옵션 (시간 허용 시)
- FEAT-2: 토스페이먼츠 결제 연동
- 플랜 비교 페이지 UI

### 5.3 Non-goals (명시적 제외)
- 커스텀 도메인 연결 (인프라 복잡도 높음)
- 연간 결제 (초기에는 월간만)
- Enterprise 플랜 (수요 확인 후)
- 환불 자동화 (수동 처리로 시작)

## 7. 해결 가설과 실험

### 7.1 가설

> 만약 무료 플랜에 적절한 제한(프로젝트 3개, 내보내기 10회/일)을 두면,
> 더 많은 기능이 필요한 사용자의 5%가 유료 전환할 것이다.
> 왜냐하면 핵심 가치를 경험한 후 추가 니즈가 생기기 때문이다.

### 7.2 실험 루프

| 단계 | 내용 |
|------|------|
| 실험 | 플랜 제한 시스템 도입 (결제 연동 전) |
| 관측 | 제한 도달 메시지 노출 횟수, 플랜 페이지 방문 수 |
| 학습 | 제한에 도달하는 사용자 비율, 업그레이드 의향 |
| 다음 | 의향이 충분하면 결제 시스템 연동 |
