## 주요 의사결정

### DEC-001: 백엔드 선택 - Supabase

**결정**: Firebase 대신 Supabase 선택

**이유**:
- PostgreSQL 기반 (관계형 데이터에 적합)
- 실시간 기능 내장
- Google OAuth 지원
- 오픈소스 + 무료 티어 충분

**대안 검토**:
- Firebase: NoSQL, 경험 있음 → 관계형 데이터에 덜 적합
- 자체 서버: 개발/운영 부담 큼

---

### DEC-002: MVP 범위 - 추가 레이아웃 우선

**결정**: Phase 1에서 6종 레이아웃 추가에 집중

**이유**:
- 현재 가장 큰 한계점 (5종만 존재)
- 백엔드 없이도 즉시 가치 제공
- 사용자 경험 직접 개선

**포함 레이아웃**:
1. Gallery - 이미지 그리드
2. Timeline - 시간순 스토리
3. Cards - 카드형 나열
4. Quote - 인용문 강조
5. Stats - 숫자/통계
6. Video Hero - 비디오 배경

---

### DEC-003: 미디어 저장 정책

**결정**: 이미지/GIF 중심, 비디오 제한적

**이유**:
- 스토리지 비용 관리
- 이미지만으로 대부분 시나리오 커버
- 비디오는 용량/대역폭 부담

**정책**:
- 이미지: PNG, JPG, WebP (최대 5MB)
- GIF: 최대 10MB
- 비디오: Phase 1에서는 로컬만, Phase 2에서 제한적 클라우드

---

### DEC-004: 협업 범위

**결정**: 실시간 동시 편집 대신 편집 권한 공유

**이유**:
- 실시간 협업은 복잡도 높음 (CRDT 등)
- 1인 개발 제약
- 편집 권한만으로 대부분 시나리오 충족

**Phase 3 범위**:
- 링크 공유 (읽기 전용)
- 이메일로 편집 권한 초대
- (향후) 실시간 협업 검토

---

### DEC-005: 수익화 전략

**결정**: 기본 무료 + 향후 프리미엄 고려

**이유**:
- 초기 사용자 확보 우선
- 프리미엄 기능은 사용자 피드백 후 결정

**프리미엄 후보** (미확정):
- 클라우드 스토리지 용량 확대
- 커스텀 도메인
- 팀 협업 기능
- 고급 분석/통계

## 기술적 결정

### TEC-001: 컴포넌트 구조

**결정**: 레이아웃별 컴포넌트 분리

```
components/layouts/
├── HeroLayout.tsx
├── GalleryLayout.tsx
├── TimelineLayout.tsx
└── ...
```

**이유**:
- 각 레이아웃 독립적 개발/테스트
- 코드 가독성 향상
- 유지보수 용이

---

### TEC-002: 상태 관리

**결정**: Phase 1은 useState 유지, Phase 2에서 커스텀 훅 추가

**이유**:
- 현재 구조 단순하고 잘 작동
- Supabase 연동 시 훅으로 추상화
- 불필요한 복잡도 방지

---

### TEC-003: 타입 확장 전략

**결정**: Section 인터페이스에 옵셔널 필드 추가

```typescript
interface Section {
  // 기존 필드...
  galleryImages?: GalleryImage[];
  timelineItems?: TimelineItem[];
  // ...
}
```

**이유**:
- 기존 코드 호환성 유지
- 점진적 마이그레이션 가능
- 레이아웃별 필요 데이터만 사용
